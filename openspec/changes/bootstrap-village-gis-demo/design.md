## Context

QYCQ 项目当前只有一份村庄 Excel 数据，缺少可直接汇报的前端产品壳和稳定的数据消费边界。本次变更需要在没有真实后端、没有最终地图样式源、且原始数据质量暂不稳定的前提下，先搭出一套可演示、可扩展的 GIS 前端骨架。

当前约束如下：

- 前端技术栈固定为 React + TypeScript + Tailwind + MapLibre。
- 地图样式源后续再配置，因此地图页必须支持本地 fallback style。
- 原始 Excel 不做重清洗、不做字段归一，只做最薄的 mock 转换。
- 未来数据库会引入 `primaryid`，前端当前就必须围绕 `primaryId` 建立实体关联。
- 领导演示比复杂业务更重要，因此页面需要优先强调“看得懂”和“可讲解”。
- 响应式设计只按横屏/竖屏切换页面骨架，不按宽度断点切结构。

主要干系人包括：

- 项目负责人：需要向领导快速解释项目目标、地图能力和后续扩展方向。
- 前端负责人：后续将继续扩展页面、接入真实接口与地图源。
- 后端负责人：后续需按既定接口边界返回 `primaryid`、列表数据和 facets 数据。

## Goals / Non-Goals

**Goals:**

- 初始化一套可运行、可测试、可构建的 GIS 前端工程。
- 提供 4 个主入口：`课题简介`、`村庄地图`、`特色民俗`、`村名地理`。
- 将 `村庄地图` 实现为一个页面内的 3 种模式：`村庄检索`、`源流迁徙`、`方言分布`。
- 把所有村庄联动统一到 `primaryId`，覆盖地图、列表、详情、URL 和跨页跳转。
- 用 Python 脚本把 Excel 转成 mock JSON，并补齐稳定假坐标、搜索文本、方言分组、时间轴排序值，以及前端直接展示所需的摘要字段。
- 用集中 mapping/config 层组织颜色、路由、查询参数、时间映射、方言映射和数据源切换规则。
- 让 mock 模式与未来 API 模式通过同一 repository 契约返回相同前端领域模型。

**Non-Goals:**

- 不实现真实后端，也不锁定未来 Go + PostgreSQL 的内部实现细节。
- 不做深度数据治理、名称归一、人工纠错或专题知识抽取。
- 不处理真实多边形边界数据，首版只保证点位演示和类型兼容。
- 不建设后台录入、编辑、审核或权限管理能力。
- 不在本次变更中定义最终视觉品牌规范，只保留蓝白主题变量与集中可配能力。

## Decisions

### 1. 采用领域分层前端结构，而不是单页 demo 结构

系统将按 `app / pages / widgets / features / entities / shared` 分层组织，而不是把地图、页面和数据逻辑写在同一层。

原因：

- 当前虽然是 demo，但后续仍需继续接入真实 API 和地图源。
- `VillageRecord`、repository、mapping、runtime config 都属于跨页面复用能力。
- 先把边界分清，后面新增功能时返工成本更低。

备选方案：

- 轻量单页 demo：起步更快，但后续替换 mock、接后端和扩展页面都会挤在一起。

### 2. 用 `primaryId` 作为唯一实体键

系统内部统一使用 `primaryId` 作为村庄的唯一标识，前端不再保留第二套临时 id。

原因：

- 村名、乡镇和城市文本都可能重复或调整，无法承担长期主键职责。
- 未来后端会以 `primaryid` 为主键返回数据，前端提前统一能减少迁移成本。
- URL、地图 feature、详情查询、跨页跳转都需要同一套标识才能保证状态恢复稳定。

备选方案：

- 用 `村名 + 行政区` 组合键做前端临时关联：实现快，但后续容易与真实后端主键并存，造成双键漂移。

### 3. 用最薄 Python 脚本生成 mock，而不是在前端直接消费 Excel

原始 Excel 通过 Python 脚本转换为 `public/mock/villages.json` 和 `public/mock/facets.json`，输出结果对齐前端 `VillageRecord` 所需的摘要字段与最小派生字段。

原因：

- Excel 直接进前端会增加运行时复杂度和打包负担。
- Python 脚本更适合处理表格读取、稳定哈希和假坐标生成。
- 未来后端接管数据预处理后，前端只需切换 repository 数据源，不用改页面组件。

备选方案：

- 前端直接读 JSON/TS 常量：开发简单，但不利于和未来接口结构对齐。
- 做复杂清洗流程：超出本次 demo 目标，也会误绑未来后端实现。

### 7. 将三种运行场景表达为 runtime profile，而不是三套组件逻辑

系统对外暴露三种运行场景：`mock`、`local api`、`remote api`。内部仍只区分两类数据源：`mock` 与 `api`，并通过 runtime profile 区分本地接口和远端接口。

原因：

- 用户需要三种明确运行方式，但前端不应该因为本地接口和远端接口而分裂出两套 API 组件逻辑。
- 本地接口与远端接口的真正差异主要是运行模式和 base URL，而不是页面消费模型。
- 这种设计能同时满足“运行场景清晰”和“repository 契约统一”。

备选方案：

- 仅暴露 `mock/api` 两种运行方式：实现简单，但和已确认的 `dev/test/serve` 运行语义不完全对应。
- 为本地接口和远端接口分别写两套 repository：语义清楚，但重复且没有必要。

### 4. 用 orientation 驱动页面骨架，而不是宽度断点

系统通过 `matchMedia('(orientation: portrait)')` 和窗口高宽回退逻辑区分 `portrait` / `landscape`，并以此切换页面结构。

原因：

- 用户明确要求只按横屏/竖屏设计，不按宽度档位切结构。
- 地图页最关键的是“筛选在侧边还是抽屉里”“详情在侧边还是底部”，这本质上是方向问题。
- 方向驱动的心智更清晰，也更适合演示环境。

备选方案：

- 使用 `sm/md/lg` 断点控制页面结构：常见但与项目约束冲突。

### 5. MapLibre 使用本地 fallback style 兜底

如果未配置 `VITE_MAP_STYLE_URL`，地图仍使用本地最小 style object 渲染村庄点位。

原因：

- 当前地图源尚未就绪，但地图页必须可演示、可测试、可构建。
- 真实底图未接入时，仍需验证图层、点位、颜色和选中态。

备选方案：

- 缺少 style URL 时直接隐藏地图：会让主功能失效，不符合 demo 目标。

### 6. 将颜色、路由、查询参数、方言和时间规则集中到 mapping/config 层

所有可变规则都集中放在 `src/shared/mappings` 和 `src/shared/config`，业务组件不直接写死。

原因：

- 后续产品色、图例色、路由名、方言分组和时间映射都可能变化。
- 集中化后，修改规则不需要到页面和地图组件里全局搜改。
- 这也能作为未来前后端字段口径对齐的单一入口。

备选方案：

- 在组件中就地硬编码：短期快，但一旦颜色、映射或查询参数调整，维护成本会快速升高。

## Risks / Trade-offs

- [原始数据缺少真实坐标] → 通过稳定哈希在广东范围内生成 WGS84 假点位，并在文档中明确其演示属性。
- [`归属市|归属镇|村名` 组合未来可能冲突] → mock 阶段接受该假设；真实后端上线后以前端 adapter 直接消费后端 `primaryid`。
- [时间字段格式不统一] → 仅做最小解析，解析失败返回 `null`，前端按“时间不详”渲染，不阻断页面。
- [方言字段文本质量不稳定] → 仅做轻量关键词映射，始终保留原始字段供详情查看。
- [MapLibre 首屏资源和打包体积偏大] → 本次先保证 demo 完整性，后续根据真实地图源与性能压力再做分包或懒加载。
- [页面同时使用多个 orientation hook] → 首版接受轻量重复监听，保持实现简单；后续若页面复杂度继续增长，可抽成 context。

## Migration Plan

1. 初始化前端工程、依赖、测试配置和基础目录结构。
2. 生成 mock JSON，并在 `dev` 模式下跑通列表、facets 和地图点位。
3. 落地 4 个主页面与地图三模式交互，统一接入 `primaryId`。
4. 通过 repository 抽象隔离 mock / local API / remote API 三种运行模式。
5. 后续接入真实后端时，仅替换 `VITE_DATA_MODE=api` 与 `VITE_API_BASE_URL`，并保证后端返回 `primaryid` 字段。
6. 后续接入真实地图样式源时，仅补充 `VITE_MAP_STYLE_URL`，不重写地图页结构。

回滚策略：

- 若真实 API 未稳定，可立即切回 `mock` 模式继续演示。
- 若地图样式源异常，可移除 `VITE_MAP_STYLE_URL` 回退到本地 style。

## Open Questions

- 真实后端的 `primaryid` 最终采用字符串还是数值类型；当前前端会统一适配为字符串 `primaryId`。
- 未来是否需要引入真实 polygon 边界、迁徙路径或专题图层；当前类型已预留，但页面未实现。
- `特色民俗` 与 `村名地理` 后续是否需要升级为列表检索页或专题详情页；当前先以领导演示卡片页承载。
