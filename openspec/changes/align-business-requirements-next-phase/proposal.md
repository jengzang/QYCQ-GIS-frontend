## Why

当前项目已经完成“领导演示型 GIS 前端”的第一阶段骨架，但新的业务需求范围明显更大，已经覆盖：

- 更完整的多条件查询
- 统计分析与图表报表
- 数据新增/修改/删除/导入/导出与校验
- 专题能力增强
- 周边自然村等空间查询能力

同时，当前数据仍存在一个关键前提：源数据没有真实经纬度，地图点位是根据村庄文本键稳定生成的广东省范围内临时 WGS84 假点位，仅用于演示、联动和交互验证。

因此需要在 OpenSpec 中补齐两件事：

1. 明确当前系统的真实状态：它是“可运行的 demo 骨架 + 临时空间坐标方案”，不是已经完成全部业务闭环的正式系统。
2. 把新增业务需求拆成可交付阶段，避免一次性把查询、统计、后台管理、空间分析混成一个模糊目标。

## Current State Snapshot

- 当前前端技术栈：React + TypeScript + Vite + Tailwind + React Query + MapLibre
- 当前页面：`课题简介`、`村庄地图`、`特色民俗`、`村名地理`
- 当前地图能力：检索 / 时间轴 / 方言三模式，URL 驱动，`primaryId` 统一联动
- 当前 mock 数据：5319 个村庄，26 个城市，54 个乡镇，8 个方言分组，timeline 范围 1120-2003
- 当前坐标状态：无真实经纬度；使用广东省范围内的稳定假点位

## What Changes

- 将“临时坐标”状态明确纳入下一阶段需求上下文，要求后续统计、空间分析和周边查询都显式区分“假点位模式”和“真实坐标模式”。
- 将新增业务需求拆成 4 个能力域：
  - `village-query-enhancement`
  - `analytics-and-reporting`
  - `village-data-management`
  - `special-topics-and-spatial-enhancements`
- 将下一阶段交付建议拆成 3 个阶段：
  - P0 查询补全与专题增强
  - P1 统计分析与报表
  - P2 数据管理与空间服务
- 明确哪些需求可在临时坐标前提下先完成，哪些需求必须等待真实坐标。

## Delivery Recommendation

### P0: 查询补全与专题增强
先补齐查询能力和现有专题能力，因为这部分与当前 demo 架构最贴近，且不依赖真实经纬度。

包含：
- 补充民族、经济情况等筛选维度
- 支持多条件组合查询
- 强化查询结果高亮与详情表达
- 补强村俗 / 村里名人 / 村名来源等专题信息组织
- 保持现有 `primaryId` 联动主线不变

### P1: 统计分析与报表
在现有 mock 数据字段基础上，补充聚合统计、图表页和导出能力。

包含：
- 人口、性别比、民族、语言、经济情况统计
- 柱状图、饼图等常规图表
- 按筛选条件导出统计结果
- 明确“热力图仅在真实坐标可用后进入正式验收”

### P2: 数据管理与空间服务
这部分与演示前台是不同工作流，建议作为独立后台/管理域推进。

包含：
- 管理员 CRUD
- Excel/CSV 导入导出
- 去重与校验
- 周边自然村查询
- 真实坐标接入后的空间邻近分析

## Impact

- 将影响后续页面规划：预计新增 `统计分析` 页面，未来新增 `数据管理` 后台域。
- 将影响 repository / query / facets 契约：需要支持更多筛选维度与聚合接口。
- 将影响 OpenSpec 的验收语义：所有空间分析类能力需要声明对真实坐标的依赖等级。
